## ② 统一存储需求与建设思路(~5 min)

**【口播要点】** 四类场景 × 各自真实 I/O 特征(注意数据处理和推理的纠正点)→ 五个共同诉求 → "云中立=经济武器" → "数据基础层"思路。

**【配图1】** 中间 "JuiceFS 统一存储层",四周四个场景气泡,标注各自**真实 I/O 特征**:
- **训练**:大文件顺序读 + checkpoint 周期写,要**高吞吐**
- **推理**:高并发读模型权重/数据,要**低延迟 + 缓存命中**
- **数据处理**:要的是 **IO 吞吐**(注意:海量小文件天然不适合高吞吐)
- **线上 Agent**:小文件高频读写,要**在线低延迟 + 高并发稳定**

**讲稿:**

我们的存储要服务四类差异很大的场景,先把它们各自的真实诉求讲清楚——这里有两个容易想当然的点,我特别想纠正一下。

- **模型训练**:大文件顺序读、checkpoint 周期性写,要的是吞吐和高性能离线 I/O。

- **推理**:高并发读取模型权重和数据。这里 JuiceFS 给我们带来一个额外的红利——**训推一体**:我们训练产出的 checkpoint **直接落在 JuiceFS 上**,推理服务**读的是同一份数据,还能直接命中本地分布式缓存**,省掉了"导出 → 搬运 → 重新加载"这一整套流程。训练和推理共享同一个存储、同一份缓存,这是统一存储非常具体的价值。

- **数据处理**:很多人第一反应是"海量小文件、要元数据吞吐"——但**真正要解决的是 IO 吞吐**。而**海量小文件天然不适合高吞吐**:每个文件都有 open/stat/读取的固定开销,文件越碎、聚合吞吐越差。所以我们的思路是反过来的:**用 Ray Data 并行,把海量原始小文件预处理成 parquet 大文件,作为后续所有任务复用的"数据基础层"——只整体处理一次,后续多次复用。**(详见下面配图3)

- **线上 Agent 服务**:每个 Agent(我们叫 bot)有自己的工作空间,小文件高频读写,要的是**在线低延迟和高并发稳定性**。

把四类摆在一起,共同诉求有五个:**低成本、云中立、离线高性能 I/O、线上高并发访问、对象存储接口能力**。

**【配图2】** "移动计算、共享存储"经济模型:左侧各云算力价格牌(GPU/CPU),中间 JuiceFS 统一存储底座,上方箭头"算力跟着价格走、存储不动"。标题:**云中立 = 跟着最便宜的资源走**。

这里有个关键认知先抛出来:**我们追求"云中立"不是出于理念,而是出于经济**,它本质是一种**议价能力**——

- 对**离线场景**:**哪里 GPU 便宜,算力就部署到哪里**。存储统一在 JuiceFS 上,这就是"**移动计算、共享存储**"——算力在云之间漂移,数据不搬家。
- 对**在线 Agent**:**哪里 CPU VM 便宜,就部署到哪里**。今天在 GCP,明天可能是 Azure 或任意更便宜的 region。

让这套"漂移"快速落地的工程基座,是**统一用 Terraform 编排、以 K8S 为底座**——换一朵云,基础设施一键拉起,存储层零改动。

**【配图3】** Ray Data + JuiceFS 数据处理流:原始小文件 →(Ray Data 并行 map_batches,KubeRay/Slurm 上多 GPU)→ parquet "数据基础层" → 下游 embedding/检索/推荐多次复用。底部强调:**整个 `/data/srp` 以 JuiceFS 透明挂载给 RayCluster,代码只用普通 POSIX 路径**。

补充一下"数据基础层"的工程细节,因为它很能体现统一存储的价值:我们的 Ray Data 任务跑在 KubeRay 上(大模型/大数据)或 Slurm H100/H200 上(测试),把原始数据并行处理成 parquet,单文件聚合到 **10万~100万行**,大幅摊薄小文件开销。而**整个 `/data/srp` 路径是以 JuiceFS 透明挂载给 RayCluster 的**:开发机上写的代码、处理出的数据、产出的结果,全在同一份 JuiceFS 上,RayCluster 直接可见——**代码里甚至看不到 "JuiceFS" 字样,它只用普通 POSIX 路径,存储层对 Ray 完全透明**。任务还做了幂等(`_SUCCESS` 标记),处理过的不重复跑,真正做到"一次处理、多次复用"。

这里还藏着一个对团队效率至关重要的红利——**开发者体验**:对数据开发工程师和算法工程师来说,他们看到的**永远是一个 POSIX 文件系统路径**,完全不用关心底层是 R2、GCS 还是别的对象存储、不用理会各种文件系统/对象存储协议之间的差异。**开发机和集群是同一套路径,所有数据就当成一块本地盘来用。** 这正是大纲里说的"易用生态工具链"——它把"多云多后端"的复杂度,全部收敛在了存储层之下。这条可以提炼成一句话:

> **对工程师而言,最好的多云存储,是让他们感觉不到"多云",也感觉不到"存储"——只有一块永远在那儿的本地盘。**

所以我们的建设思路一句话:**统一技术栈(JuiceFS + K8S + Terraform),但按场景分治后端**。

## ③ 选型:从 GCS Fuse / S3 Fuse 到 JuiceFS(~5 min)

**【口播要点】** 两个场景选型路径不同,分开讲:离线是"调研多方案后一开始就定 JuiceFS";Agent 是"GCS Fuse 踩坑后迁到 JuiceFS"。

**讲稿:**

选型这件事,我们两个场景走的路其实不一样,分开讲更真实。

### 3.1 离线场景:调研业界主流方案后,一开始就选了 JuiceFS

离线场景因为一上来就是多云、要高吞吐,我们在建之前就系统调研了业界几类主流存储方案,按我们的核心诉求(**云中立 / 完整 POSIX / 高吞吐 / 分布式缓存 / 成本 / 运维复杂度**)逐一对比:

**【配图4】** 离线选型对比表:

| 方案 | 云中立 | POSIX | 高吞吐 | 分布式缓存 | 成本 | 运维 |
|---|---|---|---|---|---|---|
| 自建并行 FS(Lustre/BeeGFS/WekaFS) | ❌ 绑定自建集群,跨云难 | ✅ | ✅✅ 极致 | 部分 | 高(独立硬件) | 重 |
| 云托管并行/文件存储(FSx Lustre / Filestore·Parallelstore) | ❌ 锁定单云 | ✅ | ✅ | ✅ | 高 | 轻(托管) |
| 对象存储 + FUSE(S3FS/GCS Fuse/goofys) | ⚠️ GCS Fuse 锁 GCP | ❌ 不完整 | ❌ 差 | ❌ | 低 | 轻 |
| 缓存编排层(Alluxio/Fluid) | ✅ | ✅ | ✅ | ✅ | 中(需叠底层存储) | 重 |
| **JuiceFS** | **✅ 对象后端任选** | **✅ 完整** | **✅** | **✅ 内建** | **✅ 对象存储成本** | **✅ K8S CSI 生态** |

结论很清晰:**自建并行 FS 性能强但贵且绑硬件、跨云难;云托管并行 FS 省心但锁单云;裸 FUSE 便宜但 POSIX 和性能都不行;缓存编排层还要再叠一层底层存储。只有 JuiceFS 同时命中我们全部诉求——尤其是"云中立 + 完整 POSIX + 内建分布式缓存 + 用对象存储做后端"这个组合。**

### 3.2 Agent 场景:从 GCS Fuse 踩坑,迁到 JuiceFS

在线 Agent 场景,我们其实是先用了 **GCS Fuse**,踩了坑才迁到 JuiceFS 的。坑主要有五个:

**第一,POSIX 语义不完整。** rename、flock、symlink、随机写,FUSE 系要么不支持、要么语义不对,应用层各种诡异问题。

**第二——最致命——SIGKILL 会丢数据。**

**【配图5】** 时序原理图:进程 `write()` → GCS Fuse 写入本地 write-back 缓冲(异步)→ 还没 flush 到 GCS,Pod 被 OOM/SIGKILL → 缓冲数据丢失。红色标出"数据丢失窗口"。

展开讲下,因为它是架构性的。GCS Fuse 的写是**异步 write-back**:进程 `write()` 返回成功,数据其实还在本地缓冲,过一会儿才异步刷到 GCS。但 **GCS 是对象存储,不是文件系统**,没有原子写、没有真正的 fsync 语义。一旦 Pod 被 OOM kill / SIGKILL,那些"看起来写成功"的数据就**永久丢了**——在我们 Agent 场景里直接表现为会话数据丢失。

**第三,小文件性能差、延迟高。** Agent 工作目录约 20 个 50-200KB 小文件、频繁追加写,GCS Fuse 每次 open/stat 加 50-200ms。

**第四,高并发不稳定。** 多 bot 同时读写时卡顿、报错。

**第五,被云绑定。** **GCS Fuse 只能在 GCP 内用**——连存储访问工具都被云厂商锁住,直接违背"云中立"。

**【配图6】** 时序原理图(对照配图5):JuiceFS 写路径 → 数据 chunk 先上传对象存储 → 元数据**原子提交**到元数据引擎 → 才返回成功。强调"返回成功 = 数据已落地"。

JuiceFS 怎么解的?核心在写路径:**元数据和数据分离**,数据 chunk 上传对象存储、元数据**原子提交**到独立元数据引擎之后才算写成功。"写成功"就意味着"数据真的落地了",**SIGKILL 不丢已确认的数据**。再加上**完整 POSIX**、**云中立**(对象后端随便换)、内建**分布式缓存**,和一整套生态(`juicefs sync` / S3 Gateway / CSI / 控制台)。

一句话总结:**FUSE 系是"把对象存储伪装成文件系统",JuiceFS 是"用对象存储做一个真正的文件系统"——差别就在那层独立的元数据引擎。**

## ④ 新架构:JuiceFS 在多云的部署(~7 min)

**【口播要点】** 双场景主线核心。离线 → 在线 → 决策对比 + 两条金句。

**讲稿:**

我们把 JuiceFS 用在两套差异很大的场景里。有意思的是,**同样是 JuiceFS,两边做了几乎相反的部署决策**。先看离线。

### 4.1 场景 A:离线跨云(训练 / 推理 / 数据处理)

**【配图7】** 离线跨云架构图。要素:
- 上层两朵云:**Oracle OKE(H100)** 与 **DigitalOcean DOKS(H200,NYC2/ATL1)**,跑 **Slinky Slurm + vLLM + KubeRay** 训推一体;
- 中间一条横跨两云的 **统一元数据服务**(部署在 OCI);
- 底层 **Cloudflare R2**(标注"免 egress");
- 每个 GPU 节点画出本地 **NVMe 分布式缓存**组成 cache group(标注"数据集本地化 · 首读后全命中 · 免跨云回源");
- 箭头:GPU 节点优先命中本地/邻居缓存,miss 才回源 R2。

离线要的是**算力自由漂移**:

- **算力**:Oracle 上 H100、DigitalOcean 上 H200,哪里便宜往哪加;底层统一 **Slinky Slurm on K8S** 训推一体(vLLM 推理 + Slurm 训练 + KubeRay 计算同跑一套 K8S)。
- **元数据**:**一套统一元数据服务**,跨 OCI 和 DO 共享 → 两朵云看到同一个文件系统、同一份数据,这正是"移动计算、共享存储"的落地。
- **对象存储**:用 **Cloudflare R2**。为什么?离线数据要跨云读写,**R2 免 egress 费用**,跨云流量不额外掏钱,天然契合云中立。
- **缓存**:每个 GPU 节点本地 **NVMe** 做分布式缓存,**跨节点 cache-group 共享**,数据集首读回源一次、之后基本全命中,跨云回源 R2 的延迟被它兜住。

### 4.2 场景 B:在线 Agent(ZooClaw)

**【配图8】** 在线 Agent 架构图。要素:
- 单朵云 **GCP(us-east1)**;
- **独立内网元数据服务**(3 节点 Raft,私有 VPC、内网);
- **GCS**(本云、低延迟);
- K8S 内多个 bot Pod,通过 **JuiceFS CSI** 挂同一个 **RWX PVC**,每 bot 用 **subPath** 隔离(`/zooclaw/{env}/bots/{botId}`);
- 标注"独立卷,不与离线场景联邦"。

在线 Agent 诉求完全不同:**在线、小文件、写密集、对延迟极敏感**。这里我们做了和离线**相反**的选择:

- **元数据**:不跨云联邦,在 GCP 内部起一个**独立 3 节点 Raft 元数据集群**,放私有 VPC 内网。
- **对象存储**:不用 R2,用**本云 GCS**。
- **挂载**:**JuiceFS CSI**,所有 bot 共享**一个 RWX PVC**,subPath 给每 bot 隔离目录,token 按环境限制子目录,做到文件系统级硬隔离。

### 4.3 为什么是相反的决策?

**【配图9】** 两场景决策对比表:

| 维度 | 离线跨云 | 在线 Agent |
|---|---|---|
| 首要目标 | 算力漂移、跨云共享 | 在线低延迟、高并发稳定 |
| 元数据 | **统一**(跨云共享一份) | **独立**(单云内网 Raft) |
| 对象存储 | **R2**(免 egress) | **GCS**(本云低延迟) |
| 取舍逻辑 | 跨云不掏 egress 费 | 元数据/数据贴着算力,压低 RTT |

为什么相反?**因为目标不同,最优后端就不同**:

- 离线要跨云,**R2 免 egress** 就是省钱利器;统一元数据让数据真正共享。
- 在线要低延迟。我们最早在线场景也是"OCI 元数据 + R2",结果元数据 RTT 12.7ms、跨公网,小文件写极慢,R2 偶发 30 秒 PUT 超时甚至打挂 bot。把元数据和对象存储都搬进 GCP 同一朵云内网后,延迟立刻降下来(数字下一章给)。

第一条金句:

> **云中立 ≠ 一个后端打天下。云中立的真正含义,是你有能力按场景选最优后端——而上层应用和工具链完全不用改。**

第二条金句,交了学费才悟到:

> **元数据局部性,是性能的地板。** 缓存盘再快,只要元数据要跨公网走一跳,小文件性能就被那一跳的 RTT 焊死了。

## ⑤ 实践经验:分布式缓存 / writeback / S3 Gateway(~8 min)

**【口播要点】** 干货高潮。重点讲"同一个能力,两个场景的策略差异"。

### 5.1 分布式缓存:离线"读优化" vs 在线"元数据调优"

**【配图10】** 上半:provider/consumer 模型(GPU 节点 provider 组 cache-group 共享、客户端 consumer `--no-sharing`)。下半:离线 vs 在线缓存策略对比表。

**讲稿:**

分布式缓存是 JuiceFS 的性能担当,模型是 **provider / consumer**:GPU 节点做 provider(本地 NVMe + 同一 `--cache-group`,彼此共享缓存块,一个节点拉过的数据邻居直接取);开发机/客户端做 consumer(`--no-sharing`,只读不贡献盘)。

但关键经验是:**同一个分布式缓存,我们在两个场景的优化策略完全不同。** 我去翻了两边真实的 PV mount 配置,差异很说明问题:

| | 离线(Slinky Slurm,读优化) | 在线 Agent(写优化 + 元数据调优) |
|---|---|---|
| 缓存容量 | `cache-size=-1`(用满整盘) | `cache-size=600000`(双 NVMe ~600GB) |
| 留白 | `free-space-ratio=0.3`(slurm 0.1) | `free-space-ratio=0.2` |
| 元数据 TTL | **默认(不调)** | **attr/entry/dir-entry-cache=3600** |
| writeback | **不开** | **开** |
| 特殊变体 | 流式只读用 **`cache-size=0`(nocache)** | `prefetch=3`、`open-cache`、`get/put-timeout=10` |
| 共性 | `cache-group`、`buffer-size=1024`、`max-uploads=100` | 同左 |

- **离线是"读优化"**:数据是相对静态的大文件数据集(parquet、模型权重),读密集。所以策略是**全盘缓存 + peer 共享**,数据集只首读回源一次、之后全命中;甚至对纯流式只读的场景直接用 `cache-size=0` 的 nocache 变体省盘。**离线根本没碰 attr-cache TTL**——数据静态、元数据压力小,没必要。

- **在线是"写优化 + 元数据调优"**:这里有个**反直觉的坑**——默认 `attr-cache` TTL 很短(1 秒),我们一开始发现分布式缓存几乎不工作、peer 间流量接近 0。排查发现:**TTL 太短,元数据不停失效,把 peer cache 读一直打断**。把 `attr/entry/dir-entry-cache` 从 1s 提到 **3600s** 后,分布式缓存立刻解锁,peer 间稳定跑到 **7-15 MB/s**。代价是元数据最长 1 小时可见延迟——这在"每个 bot 只写自己子目录、无跨实例热点元数据"的访问模式下完全可接受。

**所以这页的 takeaway:分布式缓存不是"开了就完事",得按场景调——离线调容量与共享,在线调元数据 TTL。**

顺带一提数据导入:我们用 `juicefs sync` 多 worker 并行(`--max-uploads=600 -p 300`,多台 GPU 当 worker),一次生产迁移 **100 万对象、38 分钟**搞定。

### 5.1.1 离线性能实测

**【口播要点】** 给离线补一组实测数字,重点讲写吞吐 ~4× 和缓存命中读 7-8 GB/s。

**【配图(离线性能)】** 对比表(蓝图风,见 PPT 离线性能页):

| 测试项(JuiceFS on R2 · 离线) | 无优化基线 | 优化后(分布式缓存 + 调参) |
|---|---|---|
| 顺序写 · 大块 | ~231 MB/s | **~714 MB/s** |
| 顺序写 · 大批量 20-50GB | ~256-265 MB/s | **840 MB/s ~ 1.1 GB/s** |
| 顺序读 · 分布式缓存命中 | — | **6.7 ~ 7.8 GB/s** |
| 顺序读 · 冷读回源 R2 | — | ~427 MB/s |

**讲稿:**

给离线场景补几个实测数字:

- **写吞吐**:没调优之前大概 **230-250 MB/s**;启用分布式缓存、把 R2 的上传下载并发调大之后,顺序写冲到 **700+ MB/s**,大批量能到 **1.1 GB/s**,差不多 **4×**。
- **读 · 缓存命中**:命中分布式缓存时能到 **7-8 GB/s**,基本就是本地 NVMe 的速度——这就是那个跨节点共享缓存池的价值。
- **读 · 冷读回源 R2**:第一次冷读约 **427 MB/s**,印证了我们反复说的"R2 下载是首读瓶颈,缓存把它兜住"。

### 5.2 Writeback:只在 Agent 开,离线不开

**讲稿:**

Writeback 是把写先落本地缓存、再异步刷对象存储。这里要明确——**我们只在 Agent 场景开了它,离线场景没开**(我特意核对了离线所有 PV 的 mountOptions,确实没有 writeback)。

这里要先点破一个本质:**writeback 是用"数据安全"换"写吞吐"**。回想③里讲的——默认情况下 JuiceFS 是"数据 chunk 先落对象存储、元数据再原子提交",所以 SIGKILL 不丢已确认数据。而**一旦开 writeback,写就先停在本地缓存盘、还没刷到对象存储,这个窗口里 crash 就会丢数据**——等于把③里那个"不丢数据"的优势主动让出去了。所以开不开,完全取决于场景能不能承受这个风险:

- **离线为什么不开**:两个原因。一是离线的写以**大文件顺序写**为主(训练 checkpoint、Ray Data 产出的 parquet),**这种顺序写直传对象存储的 IO 性能本来就够好**,不靠 writeback 提速;二是更关键的——**writeback 有 crash 丢数据的风险,而 checkpoint、数据基础层这些是宝贵且难以重算的数据,这个风险在离线场景不可接受**。所以离线宁可不要那点提速,守住数据安全。
- **Agent 为什么开**:Agent 有大量大文件产物(图像、视频、skill 包),又对在线延迟敏感,**愿意用可接受的丢数据风险换 +85% 的大文件写吞吐**(100MiB 文件 75→140 MB/s),把应用层 `write()` 和网络 I/O 解耦。

Agent 开 writeback 也踩了两个**必须讲清的坑**:

**坑一:writeback 不解决小文件 close() 延迟。** 这是常见误解。<4MiB 小文件就是一个 chunk,`close()` 是同步的、卡在对象存储 PUT 上。**小文件慢的根因是元数据 RTT,不是带宽**。writeback 帮的是大文件多 chunk 上传,帮不了小文件那一次同步 PUT——**要小文件快,得靠元数据局部性(回到那条金句),不是 writeback**。

**坑二:cache-dir 必须持久化盘,不能用 tmpfs。** 我们试过把缓存放 ramdisk,毫无提升。原因是 JuiceFS 检测到缓存后端不持久,会**静默退回同步写模式**——你以为开了 writeback,其实没生效。这种"静默降级"最坑,排查很久。

### 5.3 S3 Gateway:同一份数据,多种接口

**【配图12】** S3 Gateway 数据流图:`artifacts.xxx.ai/{botId}/...` → Cloudflare CDN → Cloudflare Worker(路径校验)→ JuiceFS S3 Gateway(:9000)→ 元数据查询 + 对象存储取 chunk。标注 single-bucket 模式、IAM 路径前缀隔离、`--keep-etag`。

**讲稿:**

S3 Gateway 把 JuiceFS 暴露成 S3 兼容接口。我们用它**对外提供 Agent 产物(artifacts)访问**:外部请求经 Cloudflare CDN → Worker 做路径校验 → 打到 S3 Gateway → Gateway 查 JuiceFS 元数据、取对象存储 chunk 返回。

实践点:
- **single-bucket 模式 + IAM 路径前缀隔离**:所有对象在一个 bucket 下,路径前缀限制只能读 `workspaces/*/artifacts/*`,bot 私有目录访问不到。
- **`--keep-etag`**:Gateway 计算 MD5 etag,配合 CDN 缓存校验。
- **多层安全**:JuiceFS token 子目录隔离 + S3 Gateway 路径限制 + Cloudflare Worker 代码级过滤,三层兜底。

**为什么这里用 S3 接口而不是 POSIX 挂载?** 因为消费方是 **CDN / Worker / Web 这类天然说 S3、HTTP 的服务**,它们不该也不需要挂一个 POSIX 文件系统。**同一份数据,内部训练用 POSIX 挂载、对外分发用 S3 接口**——这正是 JuiceFS"一份数据、多种接口"的价值。

### 5.4 性能演进:42× 是怎么来的

**【配图11】** 小文件写入速度三阶段瀑布/柱状图:Phase 1-2(OCI 元数据 + R2,缓存在启动盘)**~5-24 files/s** → Phase 3(GCP 内网元数据 + GCS + NVMe 缓存)**~1000 files/s**;标注 **~42×**、元数据 RTT **12.7ms → 5.8ms**。

**讲稿:**

把这些串起来,就是 Agent 场景的性能演进:

- 起点:OCI 元数据 + R2,跨公网。小文件写 ~5-24 files/s,元数据 RTT 12.7ms,R2 还偶发 30 秒超时打挂 bot。
- Phase 1-2:开 writeback、调 TTL 解锁分布式缓存——大文件 +85%、分布式缓存起来了,但**小文件还是慢**,因为元数据还在跨公网。
- Phase 3:元数据搬进 GCP 内网 Raft、对象存储换同区 GCS、缓存挪到 NVMe Local SSD。小文件写冲到 **~1000 files/s,约 42× 提升**;元数据 RTT 降到 **5.8ms**;GCS 同区 PUT 195-262ms、GET 38-42ms,30 秒超时彻底消失。

**42× 里最大的功臣不是缓存盘,是"把元数据搬进同一朵云的内网"。** 这就是那条金句的最佳注脚。

## ⑥ 总结 & 收益(~3 min)

**【口播要点】** 收益数字 → 回扣主线 → 前瞻战略收尾。

**【配图13】** 收益仪表盘:三个大数字 **在线 42× / 离线 ~4× / 稳定性 1年+**。

**讲稿:**

落地一年多,把收益按场景说清楚:

- **在线 Agent**:小文件写入约 **42×** 提升、大文件吞吐 **+85%**、元数据 RTT **12.7→5.8ms**;
- **离线**:启用分布式缓存 + 调参后,写吞吐 **~4×**(~250 MB/s → ~1.1 GB/s)、缓存命中读 **7-8 GB/s**;
- **稳定性**(贯穿两个场景):**1 亿+ 文件、稳定运行 1 年+、横跨三朵云**,R2 偶发 30s 超时**归零**。

> 注:成本只讲存储侧真实收益(R2 免 egress、对象存储成本 < 并行 FS,见③选型/④架构);节点成本下降来自机型/密度调整,与 JuiceFS 解耦,不在此归因。

回到开头那句话——**云中立的统一存储 + 全场景分治优化 = 低成本 + 高性能**。今天讲的两套场景、相反的后端决策,本质都在做同一件事:**让算力成本变成一个可议价、可迁移的变量,而统一存储,是这件事的地基。**

往前看,我们的离线算力会继续跟着最便宜的 GPU 漂移,在线服务也可能从 GCP 扩展到 Azure、或任意更便宜的 region——而存储层**几乎零改动**。这就是我们选择 JuiceFS 最大的理由。

## 附:Q&A 弹药

- **统一元数据跨云的网络怎么打通?** → 跨云私有网络互通,元数据走内网/受控公网,端口按官方要求放行(9400-9600 等)。
- **元数据引擎是自建还是托管?** → JuiceFS 企业版,元数据引擎由 JuiceFS 官方协助部署;离线统一集群多节点,在线 GCP 是 3 节点 Raft。
- **1 亿文件下元数据扛得住吗?** → 企业版元数据引擎 + 分片(shards);生产已验证 1 亿+ 文件、1 年+。
- **R2 vs GCS 成本到底差多少?** → R2 胜在免 egress(跨云场景);GCS 胜在同区低延迟(在线场景)。不是谁更便宜,是看场景的总成本。
- **数据处理为什么用 Ray Data + parquet,而不是直接处理原始文件?** → 海量小文件吞吐差;预处理成 parquet 大文件作"数据基础层",一次处理、多次复用,且 KubeRay 提供调度/扩缩容/failover。
- **为什么不用云厂商并行文件系统(Lustre/FSx)?** → 锁定单云、贵,不满足云中立和对象接口诉求(见③选型对比)。
