// part 01 · 需求

四类场景,四种 I/O 诉求

共同诉求:低成本 · 云中立 · 离线高吞吐 · 线上高并发 · 对象接口

我们的存储要同时服务四类差异很大的场景。先把它们各自的真实诉求讲清楚,这里有两个最容易想当然的点。

训练很直观,大文件顺序读、checkpoint 周期写,要的是吞吐。推理这里有个红利我要强调——训推一体:训练产出的 checkpoint 直接落在 JuiceFS 上,推理服务读的是同一份数据、还能直接命中本地缓存,省掉了"导出、搬运、重新加载"一整套流程。

数据处理是第一个反直觉点:很多人以为是"海量小文件、要元数据吞吐",但真正要解决的是 IO 吞吐,而海量小文件天然不适合高吞吐——每个文件都有固定开销,越碎越慢。所以我们的思路是反过来的,这页下面会讲。

最后是线上 Agent,小文件高频读写,要在线低延迟和高并发稳定。把四类放一起,共同诉求就是底下这五个词。

// part 01 · 建设思路

云中立不是理念,是议价能力

🖥️ 离线:跟着最便宜的 GPU 走

哪里 GPU 便宜,算力就部署到哪里。存储统一在 JuiceFS 上 = "移动计算、共享存储"。

💻 在线:跟着最便宜的 CPU VM 走

哪里 VM 便宜就部署到哪里。今天 GCP,明天可能 Azure 或任意更便宜的 region。

工程基座:统一 Terraform 编排 + K8S → 换云一键拉起,存储层零改动。

这页是今天的认知地基,我想先把它立住。我们追求"云中立",不是出于理念洁癖,而是出于经济——它本质是一种议价能力。

对离线场景,核心逻辑很简单:哪里 GPU 便宜,我就把算力部署到哪里。因为存储统一在 JuiceFS 上,这就变成了"移动计算、共享存储"——算力可以在云之间漂移,数据不用搬家。

对在线 Agent,逻辑一样:哪里 CPU VM 便宜就去哪里。今天我们在 GCP,明天完全可能是 Azure。

而让这套"漂移"能快速落地的,是我们统一用 Terraform 编排、以 K8S 为底座。换一朵云,基础设施一键拉起,存储层几乎零改动。所以记住这句话——云中立,是把算力成本变成一个可议价、可迁移的变量。

// part 01 · 数据处理思路 + 开发者体验

数据基础层 + 透明的 POSIX 体验

🔁 处理一次,复用多次

Ray Data 并行(KubeRay / Slurm)把海量小文件预处理成 parquet 大文件(单文件 10万~100万行),做下游 embedding/检索/推荐的"数据基础层"。

🗂️ 整个 /data/srp 透明挂载

代码里看不到 "JuiceFS" 字样,只用普通 POSIX 路径。开发机和集群同一套路径,所有数据当成一块本地盘。

对工程师而言,最好的多云存储,是让他们感觉不到"多云"、也感觉不到"存储"——只有一块永远在那儿的本地盘。

接着上一页那个反直觉点——数据处理。我们的思路是:用 Ray Data 并行,把海量原始小文件预处理成 parquet 大文件,单文件聚合到十万到一百万行,大幅摊薄小文件开销。这批 parquet 就成了后续所有任务复用的"数据基础层",只整体处理一次,后面 embedding、检索、推荐多次复用。

右边这点对团队效率特别关键——开发者体验。整个 /data/srp 是以 JuiceFS 透明挂载给 Ray 集群的。开发机上写的代码、处理的数据、产出的结果,全在同一份 JuiceFS 上。代码里甚至看不到 JuiceFS 字样,它只用普通 POSIX 路径。

对数据工程师和算法工程师来说,他们看到的永远是一个 POSIX 路径,不用关心底层是 R2 还是 GCS、不用理会任何协议差异。开发机和集群同一套路径,所有数据就当成一块本地盘。这就是底下这句话——最好的多云存储,是让人感觉不到多云、也感觉不到存储。

// part 02 · 离线选型

离线:调研业界主流方案后,一开始就选了 JuiceFS

方案云中立POSIX高吞吐分布式缓存成本/运维
自建并行 FS(比如 Lustre)❌ 绑硬件✅✅✅部分贵 / 重
云托管并行 FS(比如 FileStore)❌ 锁单云✅✅✅贵 / 轻
对象存储 + FUSE(S3FS/GCS Fuse)⚠️ 锁云❌❌❌便宜 / 轻

选型这件事,我们离线和在线走的路不一样,分开讲更真实。先说离线。

离线一上来就是多云、要高吞吐,所以我们在建之前就系统调研了业界几类主流方案,按我们的核心诉求逐一对比,就是这张表。

结论很清晰:自建并行文件系统性能最强,但贵、绑硬件、跨云难;云托管并行 FS省心但锁单云、贵;裸 FUSE便宜但 POSIX 和性能都不行;缓存编排层还得再叠一层底层存储,运维重。

只有 JuiceFS 同时命中我们全部诉求——尤其是"云中立加完整 POSIX 加内建分布式缓存加用对象存储做后端"这个组合,正好是别人都缺一块的地方。所以离线我们没怎么纠结,一开始就定了它。

// part 02 · Agent 选型

Agent:从 GCS Fuse 踩坑,迁到 JuiceFS

💥 SIGKILL 丢数据

异步 write-back,数据还在本地缓冲就被 OOM kill → 永久丢失。对象存储没有 fsync 语义。

🐌 小文件高延迟

每次 open/stat 加 50-200ms;POSIX 语义不全(rename/flock/symlink)。

🔒 被云绑定

GCS Fuse 只能 GCP 内用 + 高并发不稳定 → 直接违背云中立。

在线 Agent 这边,我们其实是先用了 GCS Fuse,踩了坑才迁到 JuiceFS 的。坑主要是这三类。

最致命的是左边这个——SIGKILL 会丢数据,而且是架构性的。GCS Fuse 的写是异步 write-back:你的进程 write 返回成功,数据其实还在本地缓冲,过一会儿才异步刷到 GCS。但 GCS 是对象存储、不是文件系统,没有原子写、没有真正的 fsync。一旦 Pod 被 OOM kill,那些"看起来写成功"的数据就永久丢了。在我们 Agent 场景里,直接表现为会话数据丢失。

中间是性能——Agent 工作目录有二十来个小文件、频繁追加写,GCS Fuse 每次 open、stat 都要加五十到两百毫秒,而且 rename、flock、symlink 这些 POSIX 语义还不全。

右边这个直接打在主线上——GCS Fuse 只能在 GCP 内用,连存储工具都被云锁死,加上高并发不稳定。这就是我们要换的原因。

// part 02 · 为什么 JuiceFS 能解

差别在那层独立的元数据引擎

❌ GCS Fuse:写成功 ≠ 落地

write() 返回 → 数据还在本地异步缓冲 → 刷盘前 crash → 丢失窗口。

✅ JuiceFS:写成功 = 已落地

数据 chunk 先上传对象存储 → 元数据原子提交 → 才返回成功。SIGKILL 不丢已确认数据。

FUSE 系是"把对象存储伪装成文件系统";JuiceFS 是"用对象存储做一个真正的文件系统"。

那 JuiceFS 凭什么能解决丢数据?核心就在写路径,关键是那层独立的元数据引擎。

左边再回顾一下 GCS Fuse:write 返回成功的时候,数据还在本地异步缓冲里,刷到对象存储之前有一个丢失窗口。

右边 JuiceFS 完全不一样:它把元数据和数据分开,数据 chunk 先上传到对象存储,元数据再原子提交到独立的元数据引擎,这之后才算写成功。所以"写成功"就真的意味着"数据已经落地了",SIGKILL 不会丢已经确认的数据。再加上完整 POSIX、云中立、内建分布式缓存,还有一整套生态工具。

这页可以记一句话——FUSE 系是把对象存储伪装成文件系统,JuiceFS 是用对象存储做一个真正的文件系统。差别就在那层元数据引擎。

// part 03 · 架构 A · 离线跨云

离线:多云算力漂移,统一元数据 + R2

进入架构,这是今天最想讲的部分。我们把 JuiceFS 用在两套差异很大的场景里,而且做了几乎相反的决策。先看离线。

离线要的是算力自由漂移。所以从上往下看:算力层是 Oracle 上的 H100 和 DigitalOcean 上的 H200,底层统一用 Slinky Slurm on K8S 的训推一体方案,vLLM 推理、Slurm 训练、KubeRay 计算跑同一套 K8S。

中间这条是关键——一套统一的元数据服务,跨 OCI 和 DO 共享,所以两朵云看到的是同一个文件系统、同一份数据,这就是"移动计算、共享存储"的落地。

缓存这层,每个 GPU 节点用本地 NVMe 做分布式缓存,跨节点组成一个共享的缓存池,数据集首读回源一次,之后基本全命中,跨云延迟被它兜住。最底层对象存储用 Cloudflare R2,因为离线要跨云读写,R2 免 egress,跨云流量不额外掏钱,天然契合云中立。

// part 03 · 架构 B · 在线 Agent

在线:单云贴身,独立元数据 + GCS

再看在线 Agent,诉求完全不同——在线、小文件、写密集、对延迟极度敏感。这里我们做了和离线相反的选择。

算力是单朵云 GCP,K8S 里跑很多 bot Pod,通过 JuiceFS CSI 挂同一个 RWX 的 PVC,每个 bot 用 subPath 隔离自己的目录,token 按环境限制子目录,做到文件系统级的硬隔离。

元数据这层是关键差异:不再跨云联邦,而是在 GCP 内部起一个独立的三节点 Raft 元数据集群,放在私有 VPC 内网。对象存储也不用 R2 了,改用本云的 GCS。

大家可能会问,为什么离线和在线刚好相反?这正是下一页要讲的——这背后的取舍逻辑,才是整个架构最值钱的地方。

// part 03 · 为什么相反

同一个 JuiceFS,相反的决策

维度离线跨云在线 Agent
首要目标算力漂移、跨云共享在线低延迟、高并发稳定
元数据统一(跨云共享一份)独立(单云内网 Raft)
对象存储R2(免 egress)GCS(本云低延迟)
取舍逻辑跨云不掏 egress 费元数据/数据贴着算力,压低 RTT
① 云中立 ≠ 一个后端打天下,而是有能力按场景选最优后端,上层零改动。
② 元数据局部性,是性能的地板。

为什么相反?因为目标不同,最优后端就不同。

离线要跨云,R2 免 egress 就是省钱利器,统一元数据让数据真正共享。在线要低延迟——我们最早在线场景其实也是"OCI 元数据加 R2",结果元数据 RTT 十二点七毫秒、跨公网,小文件写极慢,R2 还偶发三十秒的 PUT 超时,甚至把 bot 打挂。把元数据和对象存储都搬进 GCP 同一朵云的内网之后,延迟立刻降下来。

这里有两条金句,是我们交了学费才悟到的。第一,云中立不等于一个后端打天下,而是你有能力按场景选最优后端,而上层应用和工具链完全不用改。

第二,元数据局部性是性能的地板。你缓存盘再快,只要元数据要跨公网走一跳,小文件性能就被那一跳的 RTT 焊死了。这句话下一部分会被反复印证。

// part 04 · 实践 · 分布式缓存

同一个缓存,两套优化策略

配置离线 Slurm(读优化)在线 Agent(写优化 + 元数据调优)
缓存容量cache-size=-1(用满整盘)cache-size≈600GB(双 NVMe)
元数据 TTL默认,不调attr/entry/dir 提到 3600s
writeback不开开
特殊流式只读用 cache-size=0prefetch / open-cache / timeout=10

模型是 provider/consumer:GPU 节点组 cache-group 共享缓存块,客户端 --no-sharing 只读。
反直觉坑:attr-cache TTL 太短会"杀死"分布式缓存 → 1s 提到 3600s,peer 间稳定 7-15 MB/s。

进入干货。分布式缓存是 JuiceFS 的性能担当,模型是 provider/consumer:GPU 节点做 provider,本地 NVMe 加同一个 cache-group,彼此共享缓存块,一个节点拉过的数据邻居直接取;开发机做 consumer,加 no-sharing,只读不贡献盘。

但关键经验是:同一个缓存,我们两个场景的策略完全不同。我特意去翻了两边真实的 PV 配置,差异很说明问题。

离线是"读优化":数据是相对静态的大文件数据集,所以全盘缓存加 peer 共享,只首读回源一次、之后全命中,纯流式只读甚至用 cache-size=0 省盘。离线根本没碰 attr-cache TTL,因为数据静态、元数据压力小。

在线是"写优化加元数据调优",这里有个反直觉的坑——默认 attr-cache TTL 只有一秒,我们一开始发现分布式缓存几乎不工作,peer 间流量接近零。排查发现是 TTL 太短、元数据不停失效,把 peer cache 读一直打断。提到三千六百秒后,缓存立刻解锁,peer 间稳定跑到七到十五 MB 每秒。所以缓存不是开了就完事,得按场景调。

// part 04 · 实践 · 离线性能实测

离线实测:写吞吐 ~4×,缓存命中读 7-8 GB/s

测试项(JuiceFS on R2 · 离线)无优化基线优化后(分布式缓存 + 调参)
顺序写 · 大块~231 MB/s~714 MB/s
顺序写 · 大批量 20-50GB~256-265 MB/s840 MB/s ~ 1.1 GB/s
顺序读 · 分布式缓存命中—6.7 ~ 7.8 GB/s
顺序读 · 冷读回源 R2—~427 MB/s

写吞吐 ~250 MB/s → ~1.1 GB/s · 缓存命中打满本地 NVMe · 冷读回源是首读瓶颈,缓存兜住

给离线场景补几个实测数字,正好呼应前面分布式缓存那页。

先看写:没调优之前大概两百三四十 MB 每秒;启用分布式缓存、把 R2 的上传下载并发调大之后,顺序写冲到七百多 MB 每秒,大批量能到一点一个 GB 每秒,差不多四倍。

读这边更明显:命中分布式缓存的时候能到七到八个 GB 每秒,基本就是本地 NVMe 的速度——这就是那个跨节点共享缓存池的价值。而第一次冷读、要回源 R2 的时候大概四百多 MB 每秒,正好印证我们反复说的那句——R2 下载是首读瓶颈,缓存把它兜住。

所以离线和在线的故事是对称的:在线靠"把元数据搬进内网"提速,离线靠"分布式缓存池"提速,底层都是同一个 JuiceFS。

// part 04 · 实践 · writeback

Writeback = 用数据安全换写吞吐

🟢 Agent:开

大文件产物多、延迟敏感,愿意用可接受的丢数据风险换 大文件吞吐 +85%(75→140 MB/s)。

🔵 离线:不开

大文件顺序写 IO 本就够好;checkpoint/数据基础层难重算,crash 丢数据风险不可接受。

两个坑:① writeback 不解决小文件 close() 延迟(那是元数据 RTT bound);② cache-dir 必须持久化盘,tmpfs 会静默退回同步写。

第二件套,writeback。先点破本质:writeback 是用数据安全换写吞吐。回想刚才——默认 JuiceFS 是数据先落对象存储、元数据再原子提交,所以不丢数据;而一旦开 writeback,写就先停在本地缓存盘、还没刷到对象存储,这个窗口里 crash 就会丢。等于把"不丢数据"那个优势主动让出去了。

所以开不开完全看场景能不能承受这个风险。Agent 开:它大文件产物多、又延迟敏感,愿意用可接受的风险换大文件吞吐 plus 85%。离线不开:一是大文件顺序写本来就够快,二是更关键——checkpoint、数据基础层这些数据宝贵且难重算,crash 丢数据在离线不可接受,宁可不要那点提速。

Agent 开了也踩两个坑:第一,writeback 不解决小文件 close 延迟,小文件慢的根因是元数据 RTT、不是带宽,这又回到那条金句;第二,cache-dir 必须是持久化盘,我们试过用 ramdisk,JuiceFS 检测到不持久会静默退回同步写,你以为开了其实没生效,排查很久。

// part 04 · 实践 · S3 Gateway

一份数据,多种接口

# 对外提供 Agent 产物(artifacts)访问 artifacts.xxx.ai/{botId}/... → Cloudflare CDN → Cloudflare Worker # 路径校验 → JuiceFS S3 Gateway # single-bucket + IAM 路径隔离 + --keep-etag → 元数据查询 + 对象存储取 chunk

为什么不用 POSIX 挂载?消费方是 CDN / Worker / Web,天然说 S3。
内部训练走 POSIX 挂载,对外分发走 S3 接口——同一份数据。

第三件套,S3 Gateway。它把 JuiceFS 暴露成一个 S3 兼容接口。我们用它对外提供 Agent 产物的访问。

看这个数据流:外部请求先到 Cloudflare CDN,再到 Cloudflare Worker 做路径校验,然后打到 JuiceFS S3 Gateway,Gateway 查元数据、取对象存储的 chunk 返回。安全上做了三层:JuiceFS token 子目录隔离、S3 Gateway 的 single-bucket 加 IAM 路径前缀限制、再加 Worker 的代码级过滤。另外用了 keep-etag,让 Gateway 算 MD5,配合 CDN 缓存校验。

关键问题是——为什么这里用 S3 接口,而不是直接 POSIX 挂载?因为消费方是 CDN、Worker、Web 这类天然说 S3 和 HTTP 的服务,它们不该也不需要去挂一个 POSIX 文件系统。

所以同一份数据,内部训练用 POSIX 挂载,对外分发用 S3 接口——这正是 JuiceFS"一份数据、多种接口"的价值。

把前面这些串起来,就是 Agent 场景的性能演进,也是今天最有冲击力的一组数字。

起点:OCI 元数据加 R2,跨公网。小文件写大概每秒二十四个,元数据 RTT 十二点七毫秒,R2 还偶发三十秒超时打挂 bot。

Phase 1 到 2:我们开 writeback、调 TTL 解锁分布式缓存,大文件吞吐上去了 85%,分布式缓存也起来了,但小文件还是慢,因为元数据还在跨公网。

Phase 3 是质变:把元数据搬进 GCP 内网的 Raft 集群,对象存储换成同区 GCS,缓存挪到 NVMe。小文件写冲到每秒一千个,大约 42 倍提升;元数据 RTT 降到五点八毫秒;三十秒超时彻底消失。

所以这页的结论一句话——42 倍里最大的功臣不是缓存盘,是把元数据搬进了同一朵云的内网。这就是"元数据局部性是性能地板"最好的注脚。
